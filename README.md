# RoadSense — Smart Road Safety & Accident Prediction Platform

A working, demo-ready slice of the platform described in your project brief:
predicts an accident risk score (0-100) for any point on the map from
weather + traffic + time + road history, shows accident hotspots, and
compares two routes' risk the way the brief's "Road A vs Road B" example
describes.

**Scope note (read this first):** your brief lists GNNs, LSTMs, CNNs,
Transformers, live traffic APIs and PostGIS. Building all of that *and*
making it demo-stable in 10 days is not realistic — most hackathon teams
that try end up with nothing working on stage. This build instead gives you
one thing that works end-to-end (RandomForest risk model + weather + route
scoring + live map), which judges can actually click through, plus a clearly
scoped stretch goal (a GNN notebook) you only touch if the core is solid
with days to spare. That trade-off is explained more in "What's real vs
simulated" below — say this out loud to your mentor rather than letting them
find out during Q&A.

## 2. Run it (first time, ~10 minutes)

You said you use VS Code, so: open the `smart-road-safety` folder in VS Code,
then use two integrated terminals (`` Ctrl+` ``, then the `+` icon for a
second one).

**Terminal 1 — backend**
```bash
cd backend
python -m venv .venv
# Windows:  .venv\Scripts\activate
# Mac/Linux: source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8000
```
Leave this running. Visit `http://localhost:8000/docs` — you should see
Swagger UI listing `/api/predict`, `/api/hotspots`, `/api/route-risk`, etc.
Try `/api/predict` there with a Delhi lat/lon to sanity check it before
touching the frontend.

**Terminal 2 — frontend**
```bash
cd frontend
npm install
npm run dev
```
Open `http://localhost:5173`. You should see a dark map of Delhi-NCR with
colored dots (hotspots), a stats sidebar, and a route-risk panel.

If the map loads but shows no dots, the backend isn't running or CORS is
blocked — check Terminal 1 for errors first.

## 3. How to use the app (for your own testing + the demo)

- **Hotspots**: colored dots on the map. Green = safe, red = very dangerous.
  Click one for its risk score and accident history.
- **Dashboard** (left sidebar): region-wide stats, a risk-by-hour-of-day
  chart (rush hour shows up as a bump), and a ranked list of the riskiest
  locations.
- **Route safety check**: click "① Click map to set start", click anywhere
  on the map, then "② Click map to set destination", click again, then
  "Compare route risk". It fetches a real route from OSRM, scores it point
  by point, and — if OSRM returns an alternative — recommends the safer one.
  This *is* the brief's Road A/Road B example, running on live-ish data.

## 4. Swapping in real data (do this by day 3-4, see plan below)

Right now `backend/app/data/accidents.csv` is **synthetic** — generated by
`generate_data.py` with realistic patterns (rush-hour traffic, monsoon rain,
winter fog, hotspots that get worse over time) but not real accidents. Judges
will ask about this; have your answer ready ("we built the full pipeline
against a synthetic dataset with the same schema as [source], so swapping in
the real data is a config change, not a rewrite").

Real sources to try:
- **data.gov.in** — search "road accident" for state/city-level datasets
- **MoRTH "Road Accidents in India"** annual reports (PDF/Excel, needs
  cleaning)
- **Kaggle "US Accidents"** dataset — much cleaner, has lat/lon/weather/time
  already, good fallback if Indian data is too messy to clean in time

To swap in real data: reshape it to the same columns as `accidents.csv`
(see the column list at the top of `train_model.py` —
`NUMERIC_FEATURES` / `BOOLEAN_FEATURES` / `CATEGORICAL_FEATURES`), save it
over `backend/app/data/accidents.csv`, then re-run:
```bash
cd backend
python app/data/generate_data.py   # skip this if using your own CSV
python -m app.ml.train_model
```
Restart uvicorn afterward to pick up the new model.

## 5. What's real vs simulated (say this explicitly to judges)

| Module in the brief | This build | Why |
|---|---|---|
| ML risk prediction | **Real** — trained RandomForestRegressor, R²≈0.93 on held-out data | scikit-learn, no shortcuts |
| Weather Analysis | **Real live API** — Open-Meteo, free, no key | doc's OpenWeatherMap needs signup+key |
| Traffic Density | **Simulated** — rush-hour heuristic | Google/HERE/TomTom traffic needs billing-enabled keys |
| Accident Hotspot Prediction | **Real** — aggregated from dataset | — |
| Route Risk Scoring | **Real** — live OSRM routing + per-point model scoring | doc's Google Maps API needs a billing key |
| Dashboard | **Real** — React + Leaflet + Recharts | — |
| GNN / LSTM / CNN / Transformer | **Not built** — out of scope for 10 days | see stretch goal below |
| PostGIS / Postgres | **Not used** — CSV + pandas | swap in later, see section 7 |

Being upfront about this table is genuinely a stronger pitch than pretending
everything in section 13-14 of the brief is live — judges have seen a hundred
overclaimed AI projects and reward teams who scoped honestly and shipped
something that works.

## 6. 10-day plan

**Days 1-2 — Get it running, make it yours**
- Run the setup in section 2. Get comfortable with the code.
- Change `BBOX` in `generate_data.py` to your own city if you don't want
  Delhi, regenerate data, retrain.
- Read `train_model.py`'s printed feature importances — memorize the top 3,
  you'll want them for Q&A ("historical accident count and traffic density
  dominate the model").

**Days 3-4 — Real data pass**
- Try to get *some* real accident data in, even for a subset (one city, one
  year). Even partially-real data changes your pitch from "simulated" to
  "hybrid: real accident history + simulated live conditions."
- If real data cleaning is eating too much time, timebox it to end of day 4
  and fall back to the synthetic set — don't let this block everything else.

**Days 5-6 — Polish the core features**
- Tune `SAMPLE_POINTS` and thresholds in `route_service.py` if route
  scoring feels off.
- Add 3-5 preset "famous risky routes" in your own city as buttons (skip
  needing to click the map during the live demo — clicking precisely on
  stage is a common hackathon failure point).
- Test on the actual wifi you'll demo on. If OSRM is blocked/slow on venue
  wifi, that's the day to find out, not demo day.

**Days 7-8 — Stretch goal (only if 1-6 are solid)**
- The Graph Neural Network is genuinely your best differentiator for
  judges who know ML, but it's also the highest-risk piece to get running.
  Treat it as an **offline notebook demo**, not a live API dependency:
  - Build a small road graph (nodes = your 40 hotspot locations, edges =
    nearest-neighbor connections) with `networkx` + `torch_geometric`.
  - Train a 2-layer GCN to predict node risk from neighboring nodes' risk +
    features — this directly demonstrates the brief's "if Road A is
    blocked, Road B's risk rises" idea.
  - Show it as a slide/notebook output ("here's the GNN proving connected
    roads share risk") rather than wiring it into the live app under time
    pressure.
- If you don't reach this, the core app alone is a complete, honest
  submission — don't sacrifice its stability for this.

**Day 9 — Rehearse the story, not just the code**
- Practice the pitch: problem → Road A/Road B motivation → live demo
  (predict a point, compare two routes) → the honesty table in section 5 →
  future scope (section 20 of your brief: CCTV, IoT, V2I).
- Prepare for the two questions every judge asks: "Is this real data?" (see
  section 5) and "How does the model actually decide risk?" (show
  `feature_importance.json` — historical accidents, traffic, and visibility
  are your top drivers, in plain language: "places that have been dangerous
  before, that are busy right now, in bad visibility, are flagged first").

**Day 10 — Buffer**
- Something will break the night before. This day exists for that. Freeze
  new features after day 9.

## 7. Natural next upgrades (if you have time or want to keep going after)

- Swap CSV+pandas for **PostgreSQL + PostGIS** once the schema is stable —
  makes spatial hotspot queries much faster at scale.
- Replace the OSRM public demo server with a **self-hosted OSRM** instance
  (Docker image) if you outgrow the public server's rate limits.
- Add the LSTM/CNN/Transformer/GNN modules from sections 12-13 of your
  brief as separate, optional model files behind a feature flag, once the
  core system is stable — don't let research ambition destabilize a working
  demo before a deadline.

## 8. Troubleshooting

- **Frontend shows a blank map / no tiles**: check your internet connection —
  map tiles load from CARTO's CDN.
- **"Routing failed" error in the route panel**: the free OSRM demo server
  is rate-limited and occasionally slow. Wait a few seconds and retry, or
  self-host OSRM (section 7) before your demo if this happens often on your
  wifi.
- **`ModuleNotFoundError: No module named 'app'`**: run uvicorn from inside
  `backend/`, not from the repo root.
- **CORS errors in the browser console**: confirm the backend is running on
  port 8000 and the frontend's `VITE_API_URL` (if you set one) matches.
